# Getapps 设计系统

**职责：做高保真原型页面。** 从设计稿或需求描述出发，产出可运行的移动端页面。不负责制作/更新组件规范（那是 `/spec-maker` 的事）。

所有文件位于 `~/.claude/commands/ga-gh-design/references/`。

---

## 一、设计哲学

Getapps 是小米应用商店海外版，基于 HyperOS 3 设计语言。

| 特征 | 实际指导 |
|------|---------|
| 三色体系 | 绿色 #05C474（默认）、黄色 #FFE249（社区）、蓝色 #3482FF（OS 系统控件），通过 CSS 变量切换 |
| 移动端优先 | 无 :hover 只有 :active，触控区最小 44px，392px 基准宽度 |
| Token 驱动 | 所有颜色/字号/圆角/间距/阴影由 Token 定义，禁止 hardcode，明暗自动切换 |

## 二、核心铁律

以下规则在任何情况下都不可违反。

### 组件调用规则

| 情况 | 做法 |
|------|------|
| 有现成组件 | 先读决策指南（组件名.md），再从 code MD 复制代码。**两个文件必须都读**，禁止跳过决策指南直接取代码 |
| 没有现成组件 | 用基础 Token 自建（color.md / typography.md / border-radius.md / elevation.md） |
| 不确定有没有 | 先查下方「快速查找」表 |
| 组件库无此代码 | 标注 `/* 组件库无此代码 */` 并告知用户 |

> **前置条件：** 写任何代码前，如果当前上下文中主题色未明确（用户没说、Figma 没给、现有项目没有配置），**必须先询问用户确认主题色**，然后再调用组件或自建。

### 写代码前必须检索

每写一个属性都要检索对应规范，不能凭印象：

| 要写什么 | 检索哪里 | 找到 → | 没找到 → |
|---------|---------|--------|---------|
| 组件 | `preview.html` → `*-code.md` | 直接复制 | 标注"组件库无此代码"并告知用户 |
| 颜色 | `color.md` → `:root` 变量 | 用 `var(--token)` | 不可 hardcode，先确认是否需要新增 Token |
| 图标 | `icons.md` → JS 常量 | 复制 SVG | 从 Figma 导出，`fill="currentColor"` |
| 字号/字重 | `typography.md` | 用规范值 | 不可自创，先和设计确认 |
| 圆角 | `border-radius.md` | 用 8 级体系值 | 不可自创 |
| 间距 | `_index.md`「间距规范」 | 用 4px 网格值 | 不允许 2/3/5/6/7/9 等非网格值 |

### 禁止行为（Anti-patterns）

| 禁止 | 为什么 | 正确做法 |
|------|--------|---------|
| hardcode 颜色（`#fff`、`rgba`） | 暗色模式失效，主题切换失败 | `var(--token名)` |
| 自写组件代码 | 与组件库不一致，无法同步更新 | 从 code MD 复制 |
| 非 4px 间距（2/3/5/7/9px） | 破坏视觉网格，与其他元素对不齐 | 用 4 的倍数 |
| 写 `:hover` | 移动端无鼠标，永远不会触发 | 用 `:active` |
| 自制图标 | 与设计稿不一致，风格不统一 | 从 icons.md 取或 Figma 导出 |
| inline style | 无法被主题覆盖，暗色模式失效 | 写 CSS class |
| hardcode 字体名 | 字体通过全局样式统一定义 | 不写 font-family |
| 绝对定位布局 | 布局混乱，无法适配 | 用 flex + padding |

## 三、工作流

### 操作模式

| 模式 | 触发场景 | 入口 |
|------|---------|------|
| 从 Figma 还原 | 用户提供 Figma URL 或截图（**无论何时，只要出现 Figma 链接就走此模式**） | Phase 1 强制分模块流程 |
| 从描述做页面 | 用户描述想要的页面/功能 | Phase 1 从需求确认开始 |
| 迭代现有页面 | 用户说"继续"/"改一下"/"加个功能" | 直接打开项目，定位修改点 |

### Phase 1: 分析

目标：搞清楚要做什么、用什么。

**从 Figma 还原时（强制分模块流程）：**

> **铁律：禁止一次性读取整个页面的全部节点后直接写代码。必须先拆模块、逐个还原。**

#### Step 1：全局结构扫描（只读顶层）

1. 从 URL 提取 fileKey + nodeId
2. 调用 Figma API，**只取 depth=2**（顶层 + 直接子节点）
3. 输出模块拆解清单：

```
本页面包含 N 个模块：

1. [模块名]（12 个节点）— 简述（主要内容、可能对应的组件）
2. [模块名]（8 个节点）— ...
3. [模块名]（35 个节点）⚠️ 建议拆分 — ...

从哪个模块开始？（建议从顶部向下依次还原）
```

**拆分规则：depth=2 的每个直接子节点 = 一个模块。** 输出清单时标注每个模块的子节点数量。超过 20 个子节点的模块标记 ⚠️ 并建议用户拆分。用户确认是否拆分——同意则对该模块做 depth=2 二次拆解，不同意则按原样执行。

4. **等待用户确认或调整**，不可跳过

#### Step 2：逐模块深读 + 还原（循环执行）

用户选定模块后：

1. 根据该模块的节点数选择读取策略：

| 模块节点数 | 策略 | 做法 |
|-----------|------|------|
| ≤20 个节点 | 直接读取 | 一次 depth=10，直接输出核对表 |
| 20-50 个节点 | 分层读取 | 先 depth=3 看骨架，再逐子区域深读 |
| >50 个节点 | 拆子模块 | 先 depth=2 识别子区域，当作独立子模块逐个处理 |

**分层读取流程（20-50 节点）：**

```
第 1 遍：骨架（depth=3）
  → 识别子区域数量、各自的布局方式（row/column）、间距
  → 输出骨架结构给用户看

第 2 遍：逐子区域内容
  → 每个子区域的文字（字号/字重/颜色）
  → 每个子区域的图标（是否需要导出）
  → 每个子区域的交互元素

第 3 遍：装饰层
  → 背景填充、渐变、阴影、分割线、圆角
  → 这些属性最后统一补到 CSS 中
```

**拆子模块流程（>50 节点）：**

```
该模块内部较复杂，拆分为子模块处理：

模块 A（原模块）
  ├── 子模块 A-1：[名称]（xx 个节点）
  ├── 子模块 A-2：[名称]（xx 个节点）
  └── 子模块 A-3：[名称]（xx 个节点）

先处理 A-1。
```

2. 输出该模块的**线框图**（用 ASCII 描绘布局结构）：

```
## 模块 X：[名称] — 线框图

┌─────────────────────────┐
│ [图标]  标题文字          │
│         副标题           │
│              [按钮]      │
└─────────────────────────┘
```

线框图只表达**结构和位置关系**：哪些元素存在、它们怎么排列、哪些是内容、哪些是容器。不含颜色/字号等细节。

**用户确认线框图后**再输出详细核对表。如果用户说"不对"，输出以下排查提示：

```
线框图与实际不符，可能的原因：

1. 图层命名不清晰 — Figma 中该区域的 Frame/Group 命名为通用名（如 Frame 437），
   我无法判断它的功能。建议：给关键 Frame 加上语义化名称。

2. 装饰和内容混在同一层级 — 背景渐变/阴影/分割线与实际内容节点平级，
   我把装饰误判为了内容元素。建议：将装饰元素归入单独的 Group 或加 _bg_ 前缀。

3. 分组粒度问题 — 多个功能区域没有独立 Frame 包裹，
   或者一个 Frame 里包含了不相关的内容。建议：按功能区域拆分 Frame。

4. 隐藏/不可见图层干扰 — Figma 中有 hidden 图层但数据仍被读取。
   建议：删除不需要的隐藏图层。

请在 Figma 中调整后告知我，我会重新读取该模块。
```

3. 线框图确认后，输出详细的结构核对表：

```
## 模块 X：[名称]

### 节点层级
<父节点>
  └─ <子节点> (layout: row/column, padding, gap)
       └─ <孙节点> (尺寸, 颜色Token, 字号)

### 属性匹配
| 节点 | 属性 | Figma 值 | Token/CSS |
|------|------|----------|-----------|
| ... | ... | ... | ... |

### 识别到的组件
- [组件名] → 查 xx.md + xx-code.md

### 需要导出的图标/图片
- [节点ID] → xx.svg / xx.webp
```

4. **用户确认后**写该模块代码（如用户说"直接做"则跳过确认，但内部仍需核对）
5. 写完当前模块 → 输出：

```
✓ 模块 X 完成

剩余模块：
2. [模块名]
3. [模块名]

继续下一个？
```

6. 循环直到所有模块完成

#### Step 3：拼合 + 整体校验

所有模块完成后：
1. 检查模块间间距是否正确（通常 0px 或 8px）
2. 检查整体页面滚动行为
3. 检查标题栏/状态栏与内容的层叠关系
4. 暗色模式整体验证
5. 进入 Phase 2 检查清单

**从描述做页面时（强制设计决策前置）：**

> **铁律：禁止拿到描述后直接写代码。必须先确认设计方向再动手。**

#### 判断描述完整度

| 用户描述包含 | 判定 | 走哪条路 |
|-------------|------|---------|
| 页面结构 + 交互状态 + 组件选择 + 信息层级 | 完整描述 | **快速通道**：跳过 Step 2，Step 3 输出精简确认（仅列组件清单 + 自建部分），确认后直接进 Step 4 模块拆解 |
| 有结构但缺交互/状态/基调等关键维度 | 部分完整 | Step 2 只问缺失的维度，Step 3 正常输出，然后进 Step 4 |
| 只有一句话需求（如"做个设置页"） | 模糊描述 | 完整走 Step 1 → 2 → 3 → 4 |

#### Step 1：基础确认

1. 确认技术栈：H5（默认）/ Android
2. 确认主题色：绿色（GA 默认）/ 黄色（社区）/ 蓝色（OS 系统控件）
3. 确认明暗模式：亮色（默认）/ 暗色

#### Step 2：设计意图提问

向用户确认以下设计维度（用户已在描述中回答的跳过，未明确的必须问）：

| 维度 | 问题 | 选项示例 |
|------|------|---------|
| 核心任务 | 用户来到这个页面要完成什么？ | 浏览内容 / 做操作 / 做决策 / 查信息 |
| 用户状态 | 用户到达时的心理状态？ | 目标明确（快速找到） / 随便看看（需要引导） |
| 信息密度 | 页面信息量偏好？ | 高密度效率型 / 低密度沉浸型 / 中等标准型 |
| 情感基调 | 页面给人什么感觉？ | 克制工具感 / 轻松社区感 / 高级沉浸感 |

#### Step 3：输出设计决策（等待确认）

基于用户回答，输出设计方案：

```
## 设计决策

**页面类型：** [如：信息详情型 / 列表操作型 / 引导展示型]

**布局节奏：**
- 顶部区：[描述 — 用什么、占多大、什么作用]
- 中部区：[描述]
- 底部区：[描述]

**信息层级：**
- 主焦点：[什么信息最突出]
- 次要：[什么信息辅助]
- 弱化：[什么信息收起/折叠]

**组件规划：**
| 区域 | 使用组件 | 变体选择 |
|------|---------|---------|
| 顶部 | NavBar | 大标题 / 中标题 / ... |
| 内容 | List / Card / ... | ... |
| 底部 | BottomActionBar / TabBar / 无 | ... |

**需自建的部分：**
- [列出没有现成组件、需要用 Token 自建的区域]

以上方向可以吗？确认后开始搭建。
```

**等待用户确认或调整**，不可跳过。用户说"可以"/"没问题"后进入 Step 4。

#### Step 4：模块拆解（强制）

将页面拆分为独立模块，输出清单：

```
本页面拆分为 N 个模块：

1. [模块名] — 简述（使用什么组件、大致内容）
2. [模块名] — ...
3. [模块名] — ...

从哪个模块开始？
```

**等待用户确认**。

#### Step 5：逐模块执行（循环）

用户选定模块后：

1. 先输出该模块的**线框图**（ASCII 描绘布局结构）：

```
## 模块 X：[名称] — 线框图

┌─────────────────────────┐
│ [元素A]  [元素B]         │
│ [元素C]                  │
│         [元素D]          │
└─────────────────────────┘
```

表达该模块内部的元素构成和位置关系。用户确认理解正确后再写代码。

如果用户说"不对"，输出排查提示：
- 我对描述的理解可能有偏差 — 请指出哪个元素的位置或存在性不对
- 模块拆分粒度可能不合适 — 是否需要把该模块进一步拆分或与其他模块合并

2. 用户确认后，写该模块的代码
3. 完成后输出：

```
✓ 模块 X 完成

剩余模块：
2. [模块名]
3. [模块名]

继续下一个？
```

4. 循环直到所有模块完成

#### Step 6：拼合 + 整体校验

所有模块完成后：
1. 检查模块间间距是否正确（通常 0px 或 8px）
2. 检查整体页面滚动行为
3. 检查标题栏/状态栏与内容的层叠关系
4. 暗色模式整体验证
5. 进入 Phase 2 检查清单

### 搭建规则（逐模块执行时参照）

> 不是顺序阶段，而是在 Figma Step 2 / 描述 Step 5 执行每个模块时遵守的规则。

**项目初始化（第一个模块开始前执行一次）：**
1. 从 `page-template-code.md` 复制页面骨架（HTML + CSS + JS）
2. 安装 `debug-tool-code.md`（新页面必装）
3. 配置 `:root` 颜色变量（从 color.md 复制需要的 Token）
4. 在项目目录执行 `git init && git add -A && git commit -m "初始化项目"`

**组件调用（有组件时）：**
1. 读决策指南（如 `navbar.md`）→ 确定用哪个变体
2. 从 code MD（如 `navbar-code.md`）复制 HTML + CSS + JS
3. 填入具体内容：
   - 文字：用户提供的文案，原文照搬
   - 图标：从 `icons.md` 取 SVG，`fill="currentColor"`
   - 参数：按决策指南传入

**Token 自建（无组件时）：**
1. 布局：flex/grid，间距用 4px 网格（查 `_index.md` 间距表）
2. 颜色：查 `color.md`，用 `var(--token)`
3. 字号/字重：查 `typography.md`，用规范值
4. 圆角：查 `border-radius.md`，用 8 级体系
5. 阴影：查 `elevation.md`，用 4 级梯度
6. 交互：只写 `:active`，不写 `:hover`

**暗色模式：**
1. 所有颜色 Token 已在 `:root` 定义 light/dark 双值
2. 检查是否需要 `--token-rgb` 变体（用于 rgba 渐变）
3. 用 debug-tool 切换明暗验证

**模块完成后：**
1. 该模块代码可正常渲染（不依赖未完成模块）
2. 告知用户当前模块完成，列出剩余模块
3. 等待用户指示下一个模块

**版本控制：**

自动保存时机：
- 项目初始化完成后
- 每个模块完成后
- 所有模块拼合 + 检查通过后

用户随时可以说"保存一下"手动触发一次存档。

命令：`git add -A && git commit -m "[说明]"`

**用户说"回退"/"撤销"时：**
1. `git log --oneline -10` 展示版本列表
2. 用户选择要回到的版本
3. `git checkout <commit> -- . && git commit -m "回退到：[版本说明]"`

### Phase 2: 检查

目标：确保代码合规。

| 检查项 | 方法 | 不通过怎么办 |
|--------|------|-------------|
| 无 hardcode 颜色 | grep `#[0-9a-f]` / `rgba(` | 替换为 `var(--token)` |
| 间距合规 | grep 所有 px 值，确认为 4 的倍数 | 调整为最近的 4px 倍数 |
| 无 :hover | grep `:hover` | 改为 `:active` 或删除 |
| 图标规范 | grep `fill=`，确认为 currentColor | 修改 fill 值 |
| 无 inline style | grep `style="` | 改为 CSS class |
| 明暗适配 | 浏览器切换 dark mode | 检查哪些 Token 缺 dark 值 |
| 组件一致 | 对比 code MD 原文 | 回到 code MD 重新复制 |

### Phase 3: 交付

目标：产出可用的页面文件。

1. **预览**：
   - 启动本地服务器（端口 9000）
   - 浏览器打开页面
   - 生成二维码，提供手机预览链接
2. **命名检查**：
   - 文件名/class 名符合命名规则（见 `_index.md` 第六章）
3. **切图导出**（如需）：
   - 按 `_index.md` 切图规范执行
   - 从 Figma 按画框切、WEBP 3x、模块前缀命名
4. **代码整理**：
   - 删除调试代码
   - CSS 按组件分块
   - JS 逻辑清晰

## 四、快速查找

### 基础 Token

| 需要什么 | 查哪个文件 |
|---------|-----------|
| 详细规范（间距/命名/暗色/切图） | `_index.md` |
| 颜色 Token | `color.md` — 66 个 Token，`var(--Token名)` |
| 字体 | `typography.md` — 15 个 Text Style |
| 圆角 | `border-radius.md` — 8 级梯度 |
| 阴影 | `elevation.md` — 4 级投影 |

### 组件（决策 → 代码）

| 组件 | 决策指南 | 代码（复制用） |
|------|---------|--------------|
| 按钮 | `button.md` | `button-code.md` |
| 标题栏 / Tab 栏 / 状态栏 | `navbar.md` | `navbar-code.md` |
| 弹窗 | `dialog.md` | `dialog-code.md` |
| 输入框 | `input.md` | `input-code.md` |
| 列表 | `list.md` | `list-code.md` |
| 抽屉 | `drawer.md` | `drawer-code.md` |
| 分段按钮 | `segmented-controls.md` | `segmented-controls-code.md` |
| Checkbox | `checkbox.md` | `checkbox-code.md` |

## 五、启动选项

运行 `/ga-gh-design` 时，先询问用户：

### 1. 创建新项目
- `cp -r references/page-template/ ~/Documents/claude/output/项目名/`
- 浏览器打开页面
- 在 `~/Documents/claude/output/` 启动本地服务器（端口 9000）
- 生成二维码（`qrcode.png`）放进项目目录
- 提供手机预览链接：`http://本机IP:9000/项目名/index.html`
- 询问下一步：**提供 Figma URL** / **直接描述需求** / **自行开始**
- 如果用户提供 Figma URL → **强制进入 Phase 1 分模块流程**（禁止跳过模块拆解）
- 如果用户的回复中未明确主题色，再确认：**绿色**（GA 默认）/ **黄色**（社区）/ **蓝色**（OS 系统控件）

### 2. 继续迭代
- 扫描 `~/Documents/claude/output/` 下已有项目（按修改时间倒序）
- 用户选择后启动服务器、打开浏览器、提供预览链接

### 3. 查看/修改组件库
- 查阅或更新现有组件规范和代码

## 六、自动更新

每次运行 `/ga-gh-design` 时，先静默拉取最新版本：

```bash
cd ~/.claude/commands/ga-gh-design && git pull --quiet 2>/dev/null
```

## 七、版本管理

| 操作 | 命令 |
|------|------|
| 保存 | `cd ~/.claude/commands/ga-gh-design && git add -A && git commit -m "<说明>"` |
| 回退 | `git log --oneline -5` 让用户选，确认后 `git checkout <commit> -- . && git commit` |
| 历史 | `git log --oneline -20` |

完成一组完整修改后自动保存版本。
